## 1. Executive Summary

Knowledge2 ICP is functionally rich and has most of the raw ingredients for a serious Agentic GTM product: hundreds of seeded accounts, criteria versioning, source management, prospect role trees, research answers, K2 sync, and eval summaries. The current UI, however, still behaves like a technical demo console. It exposes backend/debugging machinery that a BDR should not have to understand, and it does not yet guide the user through the daily work of deciding which accounts to work, who to contact, what to say, and what to do next.

The highest-impact change is to reposition the product around a BDR operating loop: prioritize accounts, inspect account context, confirm prospects, launch outreach, capture feedback, and let K2 continuously expand/refine the ICP behind the scenes. K2 primitives, eval runs, trace links, and sync diagnostics should be admin/debug surfaces. Normal users should see outcomes: account fit, confidence, evidence, freshness, recommended action, and warnings only when those warnings change a GTM decision.

Overall assessment: the app is good enough as an internal technical demo, but not yet convenient enough for daily BDR/GTM use. The recommended next milestone is not a broad frontend rewrite. It is a targeted workflow redesign around a BDR command center, account pursuit workspace, source intelligence, structured research briefs, and a hidden admin diagnostics layer.

## 2. Target Audience & Evaluation Criteria

### 2.1 Primary users

| User | Goal | UX expectation |
|------|------|----------------|
| BDR / outbound rep | Work the best accounts first, find the right people, personalize outreach, track next steps. | Fast queue, clear priority, one-click actions, contacts and talking points close together. |
| GTM operator | Keep sources healthy, run enrichment, monitor pipeline creation, assign accounts. | Source ROI, schedule control, owner/status views, error recovery, budget visibility. |
| ICP owner | Refine ICP criteria and prompts without breaking the pipeline. | Versioned editor, impact preview, linting, examples gained/lost, rollback. |
| K2/platform operator | Debug automation and know when data/eval quality is drifting. | Admin-only diagnostics, eval failures, trace links, K2 sync status, deploy-safe gates. |

### 2.2 Evaluation heuristics

- BDR task fit: prioritization, time-to-first-action, account context, persona targeting, outreach readiness.
- Information hierarchy: whether the first viewport answers "what should I do now?"
- Data-table usability: find, compare, inspect, and act on records.
- Progressive disclosure: keep operator/debug detail available without dominating the BDR flow.
- Accessibility: focus visibility, target size, keyboard flow, screen-reader semantics, mobile touch ergonomics.
- Empty/error recovery: no-results states should suggest useful next moves.
- K2 alignment: use Metadata, Feeds, Agents, Streams, Quality, Feedback, and Evals as backend primitives; expose only user-facing outcomes and keep raw primitives in admin diagnostics.

### 3.2 Component inventory

| Category | Count | Quality | Notes |
|----------|-------|---------|-------|
| Page-level views | 9 tabs | Fair | Functional coverage is broad, but navigation is organized by system area rather than user workflow. |
| Reusable UI components | Ad hoc functions/classes | Fair | The app has consistent panels, pills, rows, metrics, and forms, but no component contract or state model. |
| Form components | Many native controls | Good foundation | Native controls help accessibility, but the setup/source/criteria forms need guided flows and clearer validation. |
| Data display components | Tables, cards, timelines, metrics | Fair | Dense and useful, but BDR action hierarchy is weak and large-list behavior is manually implemented. |
| Feedback/eval surfaces | K2 and Evals tabs | Poor for end users | These should not be primary navigation for BDR/GTM users. Keep them admin-only and translate useful signals into fit, freshness, evidence, and confidence. |

### 3.3 Design system maturity

The app has a coherent light palette and shared CSS variables. Buttons, inputs, panels, rows, tags, and focus rings are consistent enough for an internal product. The design system is still informal: no documented density modes, no role-based page templates, no icon/button taxonomy, no saved-view interaction model, and no reusable empty/error state pattern.

### 3.4 Accessibility baseline

Strengths:

- The app uses native `button`, `input`, `select`, and `textarea` controls.
- Top-level tab navigation has an ARIA label.
- Many controls have labels or `aria-label` values.
- Focus-visible styling is present and visible.
- Several async regions use live regions.

Gaps:

- Mobile stacks dense desktop controls rather than offering a mobile-native review mode.
- Some dense controls and row actions need larger touch targets and clearer icon/text affordances.
- Tab behavior is visually tab-like but should be audited against the ARIA tabs pattern if kept.
- Empty states usually state what is missing, but do not offer recovery actions.

## 8. Component & Library Recommendations

The current vanilla frontend can continue for the next milestone. The first investment should be local component contracts and Playwright coverage. If the app moves to React later, adopt proven headless primitives rather than a full visual framework.

| Package / pattern | Purpose | Adoption | License | Verdict |
|-------------------|---------|----------|---------|---------|
| Local design tokens and view templates | Stabilize spacing, density, commands, empty states, and drawers in the current static app. | Immediate | Internal | Adopt now. |
| TanStack Table | Sorting, filtering, selection, pagination, controlled state, and virtualization for large account/prospect tables. | Future React migration or substantial table rewrite | MIT, verify before adoption | Evaluate. |
| Radix Primitives | Accessible dialogs, popovers, menus, tooltips, tabs, and focus management if React is introduced. | Future React migration | MIT, verify before adoption | Evaluate. |
| Playwright visual/regression suite | Protect BDR cockpit, account workspace, criteria editor, and mobile review mode. | Existing E2E direction | Open source | Adopt now. |

Rationale:

- TanStack Table is useful because the app needs controlled sorting, filters, pagination, selection, and potentially virtualization for hundreds to thousands of rows while preserving product-specific UI.
- Radix is useful if the app adopts React because dialogs, menus, tabs, and tooltips need robust keyboard and screen-reader behavior.
- If the frontend stays vanilla, do not pull these libraries in indirectly. Build small local primitives and test them.

## 9. BDR/GTM Feature Strategy

### 9.1 Product direction

Move from "ICP data console" to "BDR workbench powered by K2." The app should feel like a daily operating system:

1. Start with a prioritized queue of accounts.
2. Explain why each account is worth working now.
3. Show the role tree and named prospects.
4. Provide evidence-backed talking points.
5. Launch or export outreach.
6. Capture outcome feedback.
7. Feed the outcome into K2 quality, feedback, metadata, and eval loops behind the scenes.
8. Continuously expand sources and criteria without exposing the primitive mechanics to BDRs.

### 9.2 K2 primitive alignment

| Product capability | K2 primitive fit | UX implication |
|--------------------|------------------|----------------|
| Ongoing source discovery | Feeds, Streams, Agents | Users see source freshness, newly found accounts, and source yield; stream/agent mechanics stay hidden. |
| Account qualification state | Metadata, Quality, Feedback | Users see fit tier, reason, evidence coverage, owner, status, and next action; metadata mechanics stay hidden. |
| Prospect role tree | Metadata, structured extraction, Agents | Users see named prospects and missing roles; extraction/debug details stay hidden. |
| Research answers | Search/RAG, Metadata, Traces | Users see citations, confidence, and save-to-account actions; raw trace links stay admin-only. |
| Criteria refinement | Metadata, Evals, Gold labels | ICP owners see account movement and examples; gold-label/eval internals stay admin-only. |
| Human feedback loop | Feedback, Quality, Gold labels | User actions become backend quality signals without requiring users to understand the feedback/eval model. |

### 9.3 Feature backlog for a BDR/GTM team

| Feature | User value | Notes |
|---------|------------|-------|
| Daily BDR cockpit | Shows the 25 accounts to work today, why they matter, and what action to take. | Use saved views, assignments, SLA, source freshness, and outcome tracking. |
| Next-best-action engine | Reduces decision load for each account. | Action should cite reason, persona, proof, objection, and suggested CTA. |
| Account pursuit workspace | Gives a complete plan for one account. | Role tree, contacts, evidence, sequence, notes, ownership, and CRM/Apollo export in one place. |
| Buying-trigger timeline | Makes "why now" concrete. | Use SERP, scraped pages, feeds, Apollo, seed metadata, and K2 evidence. |
| Structured research brief | Converts search/RAG into outreach-ready context. | Summary, signals, citations, objections, persona map, confidence, actions. |
| Source ROI dashboard | Shows which discovery sources actually create quality pipeline. | Track candidate count, qualified accounts, contacts, meetings, accepted/rejected feedback. |
| Mobile review mode | Lets reps approve/reject/save accounts quickly. | Phone-first triage, not a shrunken desktop table. |
| Outreach sequence builder | Turns qualification into GTM motion. | Draft variants, evidence snippets, approval, export, status sync. |
| Objection and competitive intel cards | Helps reps personalize without over-researching. | Pull from account evidence and prior outcomes. |
| Admin quality review queue | Lets backend/debugging users fix eval failures. | Admin-only gold labels, failed examples, trace links, criteria/prompt version, remediation action. |
| Collaboration and ownership | Prevents duplicate work across BDRs. | Assignment, comments, mentions, saved views, team queues. |
| Outcome-based criteria tuning | Learns from what sales accepts/rejects. | Tie accepted/replied/meeting outcomes back to criteria weights and search expansion. |

### 11.1 Risks

| ID | Risk | Likelihood | Impact | Mitigation |

| R-1 | The UI adds polish but does not change the BDR operating loop. | Medium | High | Measure time-to-first-action, accounts worked per session, and outcome feedback captured. |
